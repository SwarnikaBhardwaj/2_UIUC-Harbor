EXPERIENCE NOTES:
HttpResponse vs render():
- HttpResponse useful for simple responses, API endpoints, or when you need full control over HTML
- render() is much cleaner for template-based views, better separation of concerns
- We prefer render() for most cases as it's more maintainable

Base CBV vs Generic CBV:
- Base CBV gives more control but requires more boilerplate
- Generic CBV (ListView) is incredibly fast for standard list/detail pages
- Generic CBVs follow Django conventions, making code more predictable
- We'll use Generic CBVs for standard CRUD, Base CBVs for custom logic

SECTION 3
Static File Organization Choice:
Static files were placed at the app level under: listings/static/listings/

This structure follows Django best practices by namespacing static assets
inside the app directory. Using the additional “listings/” folder inside
static prevents naming conflicts if multiple apps include files with the
same name (e.g., styles.css).

The stylesheet (styles.css) is linked in base.html using:

{% load static %}
{% static 'listings/css/styles.css' %}

This ensures Django properly resolves static paths in development and
production environments.

Week 6
1. The Size vs. Quality Trade-off: Based on your 15-model test from the Hugging Face leaderboard, which size category provided the "sweet spot" for your project? Was a massive model significantly better than a smaller model for your specific task? Also, what would be the expected cost of each of these pre-trained models to run on your server based on computation requirements.
The size category that seemed the best for Harbor were the ultra-light and small range of models. Our final choice was the google/flan-t5-small which has about 80M parameters, significantly smaller than LLMs which boasts parameters in the multi-billion range. This constriction in size is reasonable and expected given that our primary needs for the model were narrow in nature (e.g. turning inputs such as title, category, price, and other small details from a user listing into a short description for the marketplace) over tasks that required wider, complex tasks requiring open ended reasoning. This model also fits our project needs because it runs on CPU so with the 80M model our set up time is a reasonable 1.5 seconds.
Although a larger model would provide us with more advanced context heavy polished responses, we find that for service/product marketplaces this would add very marginal value as people tend to have an idea of what they are shopping for and reply on title, price, pictures, and inference credibility of the seller.
This gain is further emphasized when we reflect on the cost savings our local model provides. Flan-t5-small can run on a virtual machine without requiring a dedicated GPU while large models in the billions of parameters often require much more compute resources. This would mean that larger models would require computation from cloud services from sources such as Google or Amazon that can range from 34 cents an hour to 100s of dollars. We would avoid all these costs entirely.

2. Cost & Scaling: If your application suddenly scaled to 10,000 daily users, would you stick with your local Django integration or move entirely to the API? Compare the potential server compute costs required to run your local model versus the per-token costs of an API.
If Harbor scaled to 10,000 users we may no longer rely on our current local Django integration. We would move to either a hybrid setup or API heavy setup to handle the greater volume and possible complexity in output tasks that we would need to complete (e.g. AI powered pricing suggestions based on product/demand/etc).
Our current set-up suits our current needs because it is lightweight and inexpensive, but with thousands of users we have greater confidence and the capital to move towards higher resource options. At 10,000 users our current set-up would fail leading to poor quality outputs for users or bottle necks leading to users having long wait times for a response or leading to no responsiveness at all. Moving towards a hybrid model would be invaluable to having a reliable product that users would want to return to. The cost to this could vary, however, we see through the Gemini developer API pricing that for text inputs are $0.50 per 1M tokens and $3.00 per 1M output tokens in their standard tier. Requests on Harbor are likely to be on the shorter side (most likely well under 100 words) given that processing a single post may even cost such under a fraction of a cent around 3,000 requests is when we may reach or surpass $1.00.

3. Hybrid Potential: How could your local Hugging Face model and your external API work together to minimize costs while maintaining high quality?
Harbor is a great fit for a hybrid architecture since it already separates a local Hugging Face model and an external Gemini API, and the system description includes a three-tier flow that first tries the local model, then Gemini, and then a template fallback.
A strong hybrid design may look like the following:
Local model manages simple and repetitive tasks such as drafting short listing descriptions, rewriting grammatically errored user inputs, and basic automatic categorization of listings
The Gemini API can then handle higher qualitative tasks such as providing more complex complex-aware descriptions for more complex posts, managing ambiguous or longer AI based search prompts from users to find listings they desire, and fallback tasks that local model failed to provide a validated output to.
This  architecture would ensure that we don’t overcomplicate when both simple and complex requests are met with the appropriate level of processing, ensure that users are happy with the output and we remain resource efficient.

4. Cost Comparison: Suppose you deploy a pre-trained model on Django, and you create a server of specific configurations (e.g., 16 core CPU, 16 core GPU, 64 GB RAM, 1 TB hard drive). Under what conditions it would be acceptable to use the pre-trained models vs when it would be better to switch to a paid API of comparable model size.
In this scenario hosting a pre-trained model would be appropriate if as mentioned prior the tasks is narrow and does not require advanced reasoning or high context-awareness, prompts are short and structured, we do not have excessively high traffic (e.g. thousands of users every hour), the selected model is compatible with the RAM/VRAM availability, and offline compatibility is valued. It would be better to switch to a paid API is usage became high, users desired more sophisticated outputs, or the functional base-line model was too large or slow for our own server to manage.
